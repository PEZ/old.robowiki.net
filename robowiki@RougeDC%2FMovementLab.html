Version History:

  • MC01: The same as the movement in RougeDC Gamma4. While I didn't change the movement since Alpha15, there was a score increase, probably from a bugfix I made in my generic wave code somewhere along the way.
  • MC02: Trying an all new surfing algorithm that I think should handle all waves better. Still needs work, the results were not worth recording even.
  • MC03: The same all new surfing algorithm as in MC02, however using the old danger formula from MC01, and works far better than MC02. It still performs worse than MC01 for the most part though, interestingly though, it gets a HOF score among the very highest there is. To me this indicates that it's better quality surfing but that the danger formula needs work to tune it to it probably...
  • MC04: Some tweaks to the danger formula. Seems to help some though the Bot C performance still isn't great.
  • MC05: Remove all but one tweak to the danger formula (making it pickier about how far away the DC match is). Bah... still a lot of work needed...
  • MC06: Change cluster size to 10 (from 20). Seems to have a nice effect overall.
  • MC07: First try with adding a neural net to the mix. I'm now suspecting that the improvements were mostly from just making it trickier for adaptive guns to understand my movement.
  • MC08: Attempts to make the neural net work better. Failed attempts not worth recording...
  • MC09: Replace the neural net with linear regression... doesn't seem to do much good...
  • MC10: Surf both adjusted and non-adjusted values.
  • MC11: MC06 plus one interpolated value.
  • MC12: MC06 plus a distance segment.
  • MC13: First attempt at building a flattener. Disable all stats except the flattener.
  • MC14: 50/50 balance of flattener and normal stats. Whoops... that was still pure flattener I think...
  • MC*MC2515: Fixed the 50/50 balance to work right, and increase cluster size to 20 for the flattener...
  • MC16: Switch the flattener to a movement profile system matching my gun.
  • MC17: Based on MC15. Add a "prefer newer data segment".
  • MC18: Cut advancing velocity segment in half and add an acceleration segment.
  • MC19: Remove the acceleration segment, but keep the advancing velocity cut to MC18 levels.
  • MC20: Revert advancing velocity level, but re-add the acceleration segment.'37.
  • MC21: Based on MC17. Cut distancing motivation in half.
  • MC22: Based on MC17. Double the distancing motivation.
  • MC17r: Re-test of what should have been the same as MC17. Now, did I screw something up by accident, or was there simply not enough seasons before?
  • MC17rr: MC17r done right.
  • MC23: Based on MC17rr. Increase rolling speed.
  • MC24: Based on MC17rr. Decrease flattener cluster size back to 10.
  • MC25: Based on MC17rr. Steal the forward/backward wall segments from my gun.
  • MC26: Based on MC25. Add a time-since-velocity-change segment.segment.
  • MC27: Based on MC26. Make the time-since-velocity-change relative to bullet-travel-time. Tweak the weighting of that segment.
  • MC28: Based on MC27. Add a weak acceleration segment.
  • MC29: Based on MC27. Try tweaking the time-since-velocity-change segment some more.
  • MC30: Based on MC12 (no flattener). Redesign the algorithm to get danger values. Before it was based upon an 'intuitively' created piecewise formula. Now it's based on creating a probability density estimation expressed as a mixture of gaussians, and to get the danger of a location, it integrates across the botwidth. Could use some tweaking I think, but the code and algorithm seems cleaner and more theoretically correct to me now.
  • MC31: Based on MC30. Change gaussian curve width to half for the distance between situation and double for the guess factor.
  • MC32: Based on MC30. Change gaussian curve width to half for the distance between situation and leave it the same for the guess factor.
  • MC33: Based on MC32. Use a more accurate [erf function] for finding the area under the gaussian curve. Courtesy of a nice public domain implementation of the erf function in [NISTMonte].
  • MC34: Based on MC33. Cut the gaussian curve width in half again for the distance between situations. Doesn't seem to help
  • MC35: Based on MC33. Increase the gaussian curve width by 50% for the guessfactor difference.
  • MC36: Based on MC33. Increase cluster size to 20. The overall score is lower, however I believe it to be notable that it helps against Splinter with hurting against GrubbmGrb, and helps against RaikoMicro. Besides, score losses against Waylander affect the score in this challenge far too much I think
  • MC37: Based on MC36. Add the forward-wall-distance segment from my flattener. It looks like additional segmentation doesn't help given how sparse the data is for non-flattener surfing.
  • MC38: Based on MC36. Cut the advancing-velocity segment to 1/4 strength. Doesn't seem to really do much good.
  • MC39: Based on MC36. Switch from a [normal distribution] to a [cauchy distribution] when estimating the likelihood of a wave being aimed towards a given GuessFactor. My motivation to switch, was because calculating cumulative distribution function (as is required to find the area under a given segment of curve) is too computationally time consuming for a gaussian/normal distribution, however is far faster for a cauchy distribution. Seems to work quite nicely.
  • MC40: Based on MC39. To commemorate to 40th test of the MovementLab, the cluster size is increased to 40! It may be a pun, but this was a serious test though, haha. It seems to help against RaikoMicro and Splinter but hurt against other things like GrubbmGrb.
  • MC41: Based on MC40. An odd experiment with varying smoothing width, wasn't successful.
  • MC42: Based on MC40. Use a cluster size of 30.
  • MC43: Based on MC42. Use a very very lightly weighted dimension to prefer new data. Interestingly it hurts performance against adaptive targeting.

Haha! The upcoming MC07 adds a NeuralNetwork to the mix, and results so far seem to show it as an improvement. What could it possibly offer a DC surfing? Well, The way I see it, normal DynamicClustering, is far from optimal against simple linear-targeting. The problem, is that for it to be accurate, it needs an awful lot of data samples (at least one at just about any velocity you may expect to be moving at) to produce good results, and it's inherently incapable of making guesses at gf values it has not seen before. Well, to solve that I added a simple back-propagation net. It's designed to find general trends in how the segment data tends to affect the guessfactor. Instead of just weighting results from DC lower when the situation is further from the current situation, I also use the neural net to reposition the guessfactors that the DC spits out, to be more relevant to the current situation. Wow... my bot is really tending to hybridize things all over the place. Actually, I find hybridizing far more pleasant than tuning, it's both more interesting and seems to get me results quicker. My targeting is DC/PM and my surfing is now DC/NN. I think this could likely be the first high ranking bot that uses both DC, PM, and NN learning/prediction techniques. Anyways, I'll post the MC07 results when I have 30 seasons I think. I think there's certainly room for improvement in the neural net though, such as trying out my "multiple plane regression clustering" idea to cluster multiple neural nets in order to more accurately model multi-mode targeters. -- Rednaxela

I find it interesting that MC35, which essentially increases smoothing across GFs helps results notably against Splinter and hurts against GrubbmGrb. I think this is because Splinter fires much less predictably and thus reduced smoothing means that RougeDC is more likely to try to slip tightly between predicted bullet locations where there isn't actually a lower chance of being fired at. I think this means that my cluster size is too small against Splinter, and also that I should perhaps do some research into [automatic bandwidth selection] for kernel density estimates. Also, another reason I believe my cluster size is too small, is because I currently suspect that in MC06 when I reduced it, the improvement I saw was only because the formula I was using to weight predicted points based on difference in situations was far from strict enough. In fact, I believe that when the formula to weight based on the distance between situations is perfect, then increasing cluster size is almost always the best choice. I think that now, I will try:
1) Increase cluster size
2) Make kernel density bandwidth depend on the amount of data points returned by the kd-tree (increased smoothing may help when there are very few data points so far)
3) Research automatic bandwidth selection
-- Rednaxela
